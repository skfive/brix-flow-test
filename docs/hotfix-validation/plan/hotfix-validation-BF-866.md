## 1. 개요 및 전제

### 1.1 목적

PR #283 은 이미 존재하는(또는 진행 중인) 핫픽스 변경분이다. BF-866 의 목적은 그 핫픽스 코드 자체를 새로 작성하는 것이 아니라, **이 핫픽스를 어떤 절차와 시나리오로 검증할지**를 후속 페르소나(designer/developer → reviewer → tester)가 공유할 수 있도록 소형 범위 계획 문서로 못박는 것이다.

### 1.2 전제 및 가정 (명시)

- 본 작업을 수행한 worktree 에는 GitHub CLI(`gh`) 및 GitHub 조회용 MCP 도구(`mcp__github__*`)가 주입되어 있지 않아, PR #283 의 실제 diff·파일 목록을 직접 조회하지 못했다.
- 저장소 검색(`git log`, `git branch -a`, 로컬 파일 grep) 으로도 PR #283 또는 BF-847 관련 파일을 로컬에서 찾지 못했다 — 즉 이 브랜치 시점 기준으로는 두 대상 모두 로컬에 실체가 없다.
- 따라서 본 문서는 **PR #283 의 정확한 코드 diff 내용에 의존하지 않는, 범용 핫픽스 검증 체크리스트 + 파이프라인 의존성 정의**로 작성한다. 실제 PR #283 을 처음 여는 개발자/리뷰어는 PR 본문·diff 를 확인해 아래 §3 시나리오를 실제 변경 파일에 맞게 구체화(항목 추가/삭제)해야 한다 — 본 문서는 그 구체화의 출발점이다.
- BF-847 은 기존 Epic 파일로 언급만 하며, 본 작업에서 열람·수정하지 않는다 (수용 기준의 범위 제약).

### 1.3 용어 정의

| 용어 | 정의 |
|------|------|
| 핫픽스 | 정규 Epic 플로우 밖에서 긴급하게 병합되는 소규모 수정. 회귀 위험이 있으므로 별도 검증 절차가 필요하다. |
| 검증 시나리오 | 핫픽스가 (a) 원래 문제를 실제로 고쳤는지, (b) 인접 기능을 깨뜨리지 않았는지를 확인하는 Given/When/Then 단위. |
| 파이프라인 | planner → designer/developer → reviewer → tester 순서로 자동 dispatch 되는 brix-Flow 표준 흐름. |

## 2. 검증 범위 (Scope)

### 2.1 포함

- PR #283 핫픽스 변경분에 대한 **검증 절차 정의**: 무엇을 확인해야 "핫픽스가 안전하다"고 판단할 수 있는지.
- 후속 페르소나(designer/developer, reviewer, tester)가 각자 단계에서 만들어야 할 **산출물 정의**와 저장 위치 컨벤션.
- 페르소나 파이프라인 **의존성 순서**와 각 단계의 진입 조건(entry condition)/종료 조건(exit condition).
- 실패 시나리오·edge case 목록 — 회귀 가드가 놓치기 쉬운 지점.

### 2.2 제외 (언급만, 수정 금지)

- **앱/런타임 코드** — 실제 핫픽스 코드, 테스트 코드, 설정 파일 등 어떤 실행 코드도 본 작업에서 작성·수정하지 않는다. 코드 변경은 developer 페르소나의 책임이다.
- **BF-847 등 기존 Epic 파일** — 다른 Epic 에 속한 기획/디자인/스펙 문서는 본 문서에서 참조 맥락으로 이름만 언급할 뿐, 내용을 읽거나 수정하지 않는다.
- PR #283 자체의 머지/재작성/충돌 해결 — 본 문서는 순수 계획 산출물이며 PR #283 에 대한 코드 조작 권한이 없다.

### 2.3 파일 소유권 (File Ownership)

본 작업이 만들거나 수정할 수 있는 경로는 `docs/hotfix-validation/plan/**` 뿐이다. 다른 모든 경로는 다른 페르소나의 담당 영역이므로 건드리지 않는다.

## 3. 검증 시나리오 (Given/When/Then)

핫픽스 검증은 아래 4개 카테고리로 나눈다. 개발자/리뷰어/테스터는 PR #283 diff 확인 후 카테고리별로 구체 항목을 추가한다.

### 3.1 카테고리 A — 원인 재현 및 수정 확인

- **Given** PR #283 이 고치려는 원래 버그(회귀/장애)가 재현 가능한 상태였고,
  **When** 핫픽스 변경분을 적용한 상태에서 동일한 재현 절차를 수행하면,
  **Then** 더 이상 해당 증상이 재현되지 않는다.

- **Given** 핫픽스가 특정 조건(예: 특정 입력값, 특정 라우트, 특정 타이밍)에서만 발생하던 문제를 다룬다면,
  **When** 그 경계 조건(정상 케이스 / 경계값 / 경계 밖 값)을 각각 실행하면,
  **Then** 정상 케이스와 경계값에서는 수정된 동작을, 경계 밖 값에서는 기존 동작을 유지한다.

### 3.2 카테고리 B — 인접 기능 회귀 없음

- **Given** 핫픽스가 공용 유틸리티/컴포넌트/설정을 변경했다면,
  **When** 그 유틸리티/컴포넌트를 사용하는 다른 모듈(다른 Epic 포함)의 기존 회귀 가드를 실행하면,
  **Then** 기존에 통과하던 테스트가 모두 그대로 통과한다.

- **Given** 핫픽스가 UI 변경을 포함한다면,
  **When** 변경된 화면과 그 인접 화면(같은 레이아웃/네비게이션을 공유하는 화면)을 렌더링하면,
  **Then** 레이아웃 깨짐·접근성 회귀(포커스 트랩, 대비, aria 속성 누락)가 없다.

### 3.3 카테고리 C — 정적 검증 게이트

- **Given** 저장소에 `pnpm verify`(또는 동등한 lint/typecheck/build 게이트)가 정의되어 있다면,
  **When** 핫픽스 브랜치에서 해당 게이트를 실행하면,
  **Then** 전체가 통과한다 — 실패 시 머지 차단.

- **Given** `BRIX_TEST_SCOPE=focused` 정책 하에서도 핫픽스가 공용 코드를 건드렸다면,
  **When** 개발자/테스터가 회귀 가드 실행 범위를 판단할 때,
  **Then** focused 범위로 한정하지 않고 공용 코드를 사용하는 다른 모듈의 테스트까지 함께 실행한다.

### 3.4 카테고리 D — 롤백 가능성

- **Given** 핫픽스 적용 후 예상 밖 회귀가 발견되었다면,
  **When** PR #283 을 되돌리는 revert 를 검토하면,
  **Then** 다른 미병합 변경과 충돌 없이 단독으로 되돌릴 수 있는 범위인지 reviewer 단계에서 확인 가능하다 (변경 파일 수·의존 관계가 diff 안에 국한되는지).

## 5. 페르소나 파이프라인 의존성

```
planner (BF-866, 본 문서)
   │  산출물: 검증 범위/시나리오/의존성 정의 (docs/hotfix-validation/plan/)
   ▼
designer / developer
   │  진입 조건: 본 문서의 §3 시나리오, §2 범위 제약을 확인한 상태
   │  산출물: (필요 시) 디자인 명세 + PR #283 실제 diff 반영 코드
   ▼
reviewer
   │  진입 조건: developer 가 PR 생성 (cascade 자동 dispatch)
   │  판단 기준: §3 카테고리 A~D 체크리스트 충족 여부
   │  결과: LGTM → pr-bot 자동 머지 dispatch  /  CHANGES_REQUESTED → developer 재dispatch
   ▼
tester
   │  진입 조건: PR merged (cascade 자동 dispatch)
   │  산출물: §3 시나리오 기반 회귀 가드 실행 결과
   ▼
완료
```

- 각 화살표(`→`)는 **시스템의 자동 cascade/webhook dispatch** 로 진행되며, 페르소나가 Jira 댓글의 `@mention` 으로 다음 페르소나를 직접 호출하지 않는다 (기록·추적 목적의 mention 만 허용).
- planner 단계(BF-866)는 코드 변경이 없으므로 PR 은 시스템이 `docs/hotfix-validation/plan/` 변경분만으로 자동 생성한다.
- developer 단계에서 실제 PR #283 diff 를 열람할 때, 본 문서 §3 의 카테고리가 실제 변경 내용과 불일치하면(예: 순수 백엔드 로직 수정이라 "인접 UI 회귀" 항목이 해당 없음) 해당 항목은 스킵하고 PR 본문/Jira 코멘트에 스킵 사유를 남긴다.

## 6. Edge Case · 실패 케이스

| # | 상황 | 대응 |
|---|------|------|
| 1 | PR #283 이 이미 머지되어 diff 를 별도로 열람할 수 없는 경우 | reviewer/tester 는 머지 커밋 기준으로 §3 시나리오를 사후 검증하고, 회귀 발견 시 별도 후속 티켓으로 처리 |
| 2 | 핫픽스가 공용 유틸리티를 변경했는데 focused 테스트 범위만 실행해 회귀를 놓치는 경우 | §3.3 카테고리 C 에 따라 공용 코드 사용처를 포함한 확장 범위 테스트를 명시적으로 실행 |
| 3 | 핫픽스 변경 범위가 BF-847 등 다른 Epic 파일과 겹치는 경우 | 본 문서/본 작업에서는 해당 파일을 수정하지 않고 이름만 언급 — 실제 충돌 해결은 developer/reviewer 가 해당 Epic 담당 페르소나와 Jira 코멘트로 조율 |
| 4 | §3 시나리오 중 일부가 PR #283 실제 diff 와 무관한 경우 | developer 가 해당 항목을 "해당 없음"으로 표시하고 사유를 PR 본문에 남김 — 시나리오를 억지로 끼워 맞추지 않는다 |
| 5 | reviewer 가 CHANGES_REQUESTED 판정 후 재수정이 §3 범위를 벗어나는 새 기능을 포함하게 되는 경우 | 핫픽스 범위를 벗어나는 변경은 별도 티켓으로 분리 — 핫픽스 PR 은 최소 범위 원칙 유지 |

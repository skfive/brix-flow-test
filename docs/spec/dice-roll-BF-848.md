## 1. 개요

### 1-1. 배경

`/demo/dice`(리포지토리 경로 `dice/`) 주사위 굴리기 SPA 는 BF-446(디자인)·BF-448(구현)·BF-450(테스트)을 거쳐 이미
동작 중이다. 그러나 원 Epic 요구사항("1~6 무작위 숫자 표시, 굴리기 버튼, 키보드 동작, 모바일 잘림 없음")은 서술형으로만
남아 있어 **검증 가능한 pass/fail 기준**이 없다. 본 문서는 이 4개 항목을 Given/When/Then 형태로 정제하고, 각 기준에
대한 구체적 검증 방법(자동화 또는 수동 QA)을 명시한다.

### 1-2. 비목표

기존 구현에 존재하는 주사위 개수 선택(1~5개)·통계 카드·히스토리·테마 토글·삭제 모달 기능 자체를 재설계하지 않는다.
이들은 §8 에 정의된 범위 밖 항목(저장·API 등)과 얽혀 있는 한 본 문서의 검증 대상에서 제외하고, 순수 시각/상호작용
계층(숫자 표시 유효성·버튼·키보드·반응형 레이아웃)만 다룬다.

### 1-3. 문서 사용 대상

- **tester**: 본 문서 §3~§6 의 Given/When/Then 을 회귀 테스트 케이스로 변환
- **developer**: 코드 변경 시 §7 Edge Case 를 회귀 체크리스트로 사용
- **reviewer**: PR 리뷰 시 §8 범위 밖 항목이 새로 도입되지 않았는지 확인

## 2. 적용 범위 요약

| 항목 | 범위 내 | 범위 밖 |
|------|---------|---------|
| 숫자 표시 | 각 주사위의 1~6 값 유효성·독립성·글리프 매핑 | 통계(합계/평균/최대) 계산 로직 |
| 굴리기 버튼 | 클릭 상호작용·연타 방지·상태 전이 | 히스토리 적재(저장) |
| 키보드 | 굴리기 트리거(Space/Enter)·포커스 이동 | 개수(1~5)·테마(T) 단축키 재정의 (기존 유지, 회귀만 없으면 됨) |
| 모바일 | 잘림(clipping) 없음 — viewport 320~414px | 태블릿/데스크톱 반응형 세부 튜닝 |
| 저장·API·이미지·애니메이션 | — | §8 참조 (전면 제외) |

## 3. AC-1: 1~6 유효 숫자 표시

### 3-1. 값 범위 유효성

```
Given 사용자가 굴리기 버튼을 클릭한다
When 굴림이 완료되어 최종 결과가 화면에 표시되면
Then 표시된 각 주사위는 1, 2, 3, 4, 5, 6 중 하나의 정수 값에 대응하는 글리프
     (⚀ ⚁ ⚂ ⚃ ⚄ ⚅ 중 하나)만 표시한다
 And 0, 7 이상, 음수, 소수, NaN, undefined 에 대응하는 표시는 발생하지 않는다
```

### 3-2. 표시 개수 일치

```
Given 주사위 개수가 N(1~5)으로 선택된 상태
When 굴리기가 완료되면
Then 화면에 표시되는 주사위 글리프의 개수는 정확히 N개다 (많지도 적지도 않음)
 And 각 글리프에는 `role="img"` + `aria-label="주사위 {값}"` 이 개별 부여된다
```

### 3-3. 값의 독립성 (편향 금지)

```
Given 여러 개의 주사위(N≥2)를 동시에 굴린다
When 각 주사위 값이 산출되면
Then 각 주사위 값은 서로 독립적으로 계산된다 (한 주사위 값이 다른 주사위 값의 범위를
     제한하거나 서로 다른 값이 되도록 강제하는 로직이 없다)
 And 모든 주사위가 동일한 값으로 나오는 결과도 유효한 결과로 허용한다
```

### 3-4. 검증 방법

- **정적 검증**: `rollOne()` 구현이 `1 + Math.floor(Math.random() * 6)` 형태(1~6 폐구간, 균등분포 산식)인지 소스 검사.
- **통계적 검증**: 자동화 테스트에서 대량 반복(예: N=2000회 이상) 호출 후 (a) 모든 결과값이 `Number.isInteger(v) && v >= 1 && v <= 6` 을 만족하는지, (b) 1~6 여섯 눈금이 각각 최소 1회 이상 출현하는지 확인한다. 특정 시드 고정은 요구하지 않는다(외부 PRNG 라이브러리 도입 금지 — §8).
- **DOM 검증**: 굴림 1회 실행 후 `#dice-box` 하위 `.dice` 요소 개수가 선택된 개수와 일치하고, 각 요소의 `aria-label` 이 표시된 글리프와 대응하는 숫자를 포함하는지 확인한다.

## 4. AC-2: 굴리기 버튼 동작

### 4-1. 기본 클릭 흐름

```
Given 페이지가 로드되어 굴리기 버튼(#btn-roll)이 활성 상태다
When 사용자가 버튼을 클릭한다
Then 버튼은 즉시 비활성화(disabled)되어 중복 클릭을 막는다
 And 굴림 진행 표시(시각 피드백)가 나타난다
 And 최종 결과가 커밋되면 버튼은 다시 활성화된다
```

### 4-2. 연타(rapid click) 방지

```
Given 굴림이 진행 중(isRolling = true)인 상태
When 사용자가 버튼을 다시 클릭(또는 Space/Enter 로 재시도)한다
Then 새 굴림은 시작되지 않는다 — 진행 중인 굴림 1회만 유효하게 완료된다
 And 클릭이 무시되었다고 해서 에러가 발생하거나 화면이 깨지지 않는다
```

### 4-3. 접근성 속성

```
Given 굴리기 버튼이 렌더링된 상태
When 버튼의 마크업을 검사하면
Then `<button type="button">` 요소이며 `aria-label="주사위 굴리기"` 를 갖는다
 And 키보드 포커스 시 `:focus-visible` 아웃라인이 시각적으로 표시된다
```

### 4-4. `prefers-reduced-motion` 사용자 동작 동등성

```
Given 브라우저/OS 설정이 prefers-reduced-motion: reduce 인 사용자
When 굴리기 버튼을 클릭한다
Then 중간 스와핑 애니메이션 단계는 생략되고 최종 결과가 즉시 표시된다
 And 버튼의 비활성화→재활성화 상태 전이(§4-1)는 동일하게 유지된다 (기능 동등성 보장)
```

### 4-5. 검증 방법

- 정적 DOM 검사: `#btn-roll` 의 `type`/`aria-label` 속성.
- 시뮬레이션(sandbox 또는 e2e): 클릭 직후 `disabled` 속성 존재 → 굴림 완료 후 `disabled` 해제까지의 상태 전이를 관찰.
- 연타 방지: 굴림 진행 중 클릭 이벤트를 추가로 발생시켜도 `state.rolls` / 표시된 히스토리 head 가 1건만 추가되는지 확인.
- reduced-motion: `window.matchMedia` mock 으로 `prefers-reduced-motion: reduce` 를 참으로 설정한 뒤 클릭 → 애니메이션 단계(`setInterval` 틱) 없이 즉시 `commitRoll` 이 호출되는지 확인.

## 5. AC-3: 키보드 동작

> 범위: **굴리기 트리거에 한정.** 개수 선택(1~5)·테마 토글(T) 단축키는 BF-446/448 산출물에 이미 정의되어 있으며
> 본 문서가 재정의하지 않는다 — 다만 아래 굴리기 트리거 동작과 충돌(중복 발동 등)이 없어야 한다는 회귀 조건만 §5-4 에 포함한다.

### 5-1. 전역 Space 트리거

```
Given 포커스가 텍스트 입력 요소(input/textarea)나 모달 위에 있지 않은 상태
When 사용자가 Space 키를 누른다
Then 페이지 스크롤 등 브라우저 기본 동작은 preventDefault 로 억제되고
 And 굴리기가 1회 트리거된다 (§4-1 과 동일한 상태 전이를 따른다)
```

### 5-2. 버튼에 포커스된 상태에서 Space/Enter

```
Given 굴리기 버튼(#btn-roll)이 키보드 포커스를 가진 상태
When 사용자가 Space 또는 Enter 를 누른다
Then 브라우저 네이티브 버튼 클릭이 1회만 발동한다
 And 전역 keydown 핸들러의 중복 트리거로 인해 굴림이 2회 발생하지 않는다
```

### 5-3. Tab 포커스 이동 및 포커스 링

```
Given 페이지가 로드된 상태
When 사용자가 Tab 키로 포커스를 이동시킨다
Then 굴리기 버튼은 Tab 순서 안에서 포커스 가능(focusable)하다
 And 포커스 시 §4-3 의 focus-visible 아웃라인이 나타난다
```

### 5-4. 굴림 진행 중 키보드 트리거 무시

```
Given 굴림이 진행 중(isRolling = true)인 상태
When 사용자가 Space 를 누른다
Then §4-2 의 연타 방지와 동일하게 새 굴림이 시작되지 않는다
```

### 5-5. 검증 방법

- 시뮬레이션: `document.dispatchEvent(new KeyboardEvent("keydown", {key:" "}))` 발생 후 굴림 상태 전이 확인 (포커스가 버튼이 아닐 때).
- 버튼 포커스 상태에서는 브라우저 네이티브 클릭 위임을 신뢰 — 전역 핸들러가 `target.tagName === "BUTTON"` 인 경우 자체 트리거를 스킵하는지 소스 검사로 확인 (중복 발동 방지 로직 존재 여부).
- Tab 포커스: `#btn-roll` 이 `disabled` 가 아닌 한 기본적으로 tabbable(별도 `tabindex` 불필요, 네이티브 `<button>`)임을 마크업 검사로 확인.

## 6. AC-4: 모바일 잘림 없음

### 6-1. 대상 뷰포트

| 대표 기기 | 뷰포트 폭 |
|-----------|----------|
| 최소 지원 폭 | 320px (iPhone SE 1세대급) |
| 소형 | 360px (다수 Android 기본) |
| 중형 | 375px (iPhone 8/12 mini) |
| 대형 | 414px (iPhone Plus/Max 계열) |

320px 미만은 §7 EC-01 에 따라 명시적으로 범위 밖이다.

### 6-2. 가로 잘림 없음 (수평 스크롤 금지)

```
Given 뷰포트 폭이 320px~414px 범위인 모바일 화면
When 주사위 개수를 1~5 각각으로 전환하고 굴리기를 실행한다
Then document.documentElement.scrollWidth 는 뷰포트 폭(window.innerWidth)을 초과하지 않는다
     (가로 스크롤바가 생기지 않는다)
 And 굴리기 버튼·주사위 박스·통계 카드·히스토리 카드의 우측 경계
     (getBoundingClientRect().right) 는 뷰포트 폭을 넘지 않는다
```

### 6-3. 요소 단위 잘림 없음

```
Given 위 뷰포트 조건
When 5개 주사위를 굴려 dice-box 에 표시한다
Then dice-box 는 flex-wrap 으로 줄바꿈되어 개별 주사위 글리프가 컨테이너 밖으로
     넘치거나 시각적으로 절단(overflow: hidden 에 의한 잘림)되지 않는다
 And 굴리기 버튼 텍스트("🎲 굴리기")는 줄바꿈되거나 말줄임 없이 완전히 표시된다
```

### 6-4. 세로 스크롤은 잘림이 아님

```
Given 뷰포트 높이가 작아 카드 4개(주사위/통계/히스토리 + kbd-hint)가 한 화면에
      다 들어가지 않는 상태
When 사용자가 페이지를 스크롤한다
Then 모든 콘텐츠는 스크롤을 통해 접근 가능해야 하며, 세로 스크롤 발생 자체는
     결함(잘림)이 아니다 — 단, 부모 요소의 overflow: hidden 등으로 인해
     콘텐츠 일부가 물리적으로 렌더링/접근 불가능해지는 경우만 결함으로 간주한다
```

### 6-5. 검증 방법

- **자동화 (권장)**: 헤드리스 브라우저(예: Playwright)로 viewport 를 320/360/375/414px 로 설정 → 개수 1~5 각각 선택 후 굴리기 → `document.documentElement.scrollWidth <= window.innerWidth` 어서션 + 핵심 요소 4종(`#dice-box`, `#btn-roll`, `.stats-card`, `.history-card`) 의 `right` 값이 `innerWidth` 이하인지 어서션.
- **수동 QA (대안)**: 브라우저 개발자 도구 반응형 모드에서 위 4개 뷰포트 프리셋으로 전환 후 육안 확인 + "Show rulers/overflow" 로 가로 스크롤바 유무 확인.
- 기존 CSS 회귀 가드(`tests/dice-ui.test.js`)는 `@media (max-width: 639px)` / `(max-width: 359px)` 규칙의 **존재**만 정적으로 검사한다 — 실제 잘림 여부(런타임 레이아웃 계산)는 검증하지 않는다. 본 AC 는 이 gap 을 메우기 위한 것이며, §9 에서 상세히 기술한다.

## 7. Edge Case 목록

| # | Edge Case | 처리 방침 |
|---|-----------|---------|
| EC-01 | 뷰포트 폭 320px 미만 (초소형 기기) | 명시적으로 범위 밖. 잘림 방지 보장 대상 아님 |
| EC-02 | 가로 모드(landscape)로 인한 낮은 뷰포트 높이 | 세로 스크롤 허용(§6-4) — 잘림 아님 |
| EC-03 | 브라우저 pinch-zoom / OS 접근성 확대 | 사용자 자발적 확대는 본 AC 의 "잘림 없음" 보장 범위 밖 |
| EC-04 | 5개 주사위 + 최소 폭 조합에서 2줄 이상으로 줄바꿈 | 정상 동작으로 허용 (flex-wrap 의도된 결과, 잘림 아님) |
| EC-05 | 굴림 애니메이션(§4-1) 진행 중 화면 회전/리사이즈 | 레이아웃 재계산 시 진행 중인 굴림이 중단되거나 결과가 소실되지 않아야 함 |
| EC-06 | Space 키를 길게 눌러 auto-repeat keydown 이 연속 발생 | §5-4 연타 방지 로직으로 굴림이 중복 시작되지 않아야 함 |
| EC-07 | `Math.random()` 이 정확히 `0` 또는 `1`에 근접한 극단값을 반환 | `1 + Math.floor(Math.random() * 6)` 산식은 `[0,1)` 구간에서 항상 1~6 정수를 산출 — 경계값 별도 처리 불필요 (수학적으로 자기충족) |
| EC-08 | 스크린리더 사용자가 굴림 결과를 인지하는 방법 | 현재는 개별 `.dice` 요소의 `aria-label`(§3-2)로만 노출되며 별도 `aria-live` 알림은 없음. 본 AC 범위에서 필수 요구사항으로 강제하지 않으나, 향후 접근성 개선 시 고려 대상으로 기록만 남긴다(비차단) |

## 8. 범위 밖 (Out of Scope) 명시

Epic 요구사항에 따라 아래 4개 항목은 본 명세 및 관련 구현 변경에서 **명시적으로 제외**한다. PR 리뷰 시 아래 항목이
새로 도입되지 않았는지 확인한다.

| 항목 | 제외 내용 | 근거 |
|------|-----------|------|
| **저장(storage)** | 굴림 결과·통계·히스토리를 위한 새로운 영속화 요구사항 추가 금지. 기존 `bf-dice-*` localStorage 키(개수/히스토리/테마)는 BF-446/448 산출물 범위이며 본 문서가 재검증·확장하지 않는다 | Epic 설명 "저장 ... 범위 밖" |
| **API / 네트워크** | 굴림 결과는 클라이언트 `Math.random()` 으로만 산출한다. `fetch`/`XMLHttpRequest`/WebSocket 등 원격 호출을 추가하지 않는다 | Epic 설명 "API ... 범위 밖", file:// CORS 안전 원칙(기존 프로젝트 전역 정책)과도 일치 |
| **외부 이미지** | 주사위 표현은 유니코드 이모지 글리프(⚀~⚅)만 사용한다. PNG/SVG/외부 CDN 이미지·아이콘 폰트를 도입하지 않는다 | Epic 설명 "외부 이미지 ... 범위 밖" |
| **애니메이션 라이브러리** | GSAP·anime.js·Lottie 등 외부 애니메이션 라이브러리를 도입하지 않는다. 굴림 시각 효과는 기존과 동일하게 CSS `@keyframes` + `setInterval` 기반 글리프 스와핑만 사용한다 | Epic 설명 "애니메이션 라이브러리 ... 범위 밖" |

> 위 4항목은 현재 구현(BF-448)이 이미 준수하고 있다 — 본 명세는 **회귀 방지 기준선**으로 이 상태를 문서화하는 것이며,
> 새로운 제약을 추가하는 것이 아니다.

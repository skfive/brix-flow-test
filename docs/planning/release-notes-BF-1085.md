# 릴리스 노트 요약 보드 기획 명세 (BF-1085)

- 대상 모듈: `release-notes` (릴리스 제목/변경 항목/영향도 입력 → 요약 카드 생성/검증 API + 요약 보드 조회)
- 관련 형제 Task: BF-1086(designer, 시안), BF-1087(developer, 구현), BF-1089(tester, 검증)
- 본 문서는 **기획 명세만** 다룬다. 코드/시안은 포함하지 않는다.
- **기술 스택 참고(가정 명시)**: 요청 설명에는 `typescript-monorepo` 마커가 있으나, 저장소 관측 규약(`base_sha` 기준)은 `vanilla-static`(패키지 매니저 npm, ESM, 정적 서빙 root `.`)이며 두 값이 불일치한다. 본 문서는 구현 프레임워크를 규정하지 않고 **동작 계약(요청/응답, 데이터 모델, 검증 규칙)만** 정의한다. 실제 구현 형태(TS 전환 여부, 모듈 구조)는 BF-1087(developer)이 저장소 실제 규약에 맞춰 결정한다.
- **독립 기능 전제**: 본 기능은 기존 라우트·데이터 스키마를 변경하지 않는 **완전히 신규 추가(additive)** 기능이다. 기존 기능과의 마이그레이션/데이터 이관은 발생하지 않는다(§10).

---

## 1. 개요

릴리스 담당자가 릴리스 제목·변경 항목 목록·영향도(중요도/사용자 영향)를 입력하면, 시스템이 이를 검증한 뒤 요약 카드(Summary Card)를 생성한다. 운영자는 요약 보드에서 생성된 카드를 중요도·사용자 영향 기준으로 필터링해 조회하고 KPI를 확인할 수 있다. 입력 제출 전 사전 점검을 위한 **검증 전용 라우트**를 별도로 제공해, 저장 부작용 없이 유효성만 확인할 수 있도록 한다.

---

## 2. 사용자 시나리오

### 시나리오 1 — 릴리스 노트 입력 및 요약 카드 생성

릴리스 담당자가 릴리스 제목, 변경 항목(1개 이상), 중요도(`importance`), 사용자 영향도(`userImpact`)를 입력해 제출한다. 모든 값이 유효하면 서버는 고유 ID와 생성 시각을 부여해 요약 카드를 생성하고 저장한다.

### 시나리오 2 — 요약 보드 조회 및 KPI 확인

운영자가 요약 보드에서 생성된 카드 목록을 중요도·사용자 영향도로 필터링해 보고, 화면 상단에서 총 카드 수·중요도별 분포·사용자 영향도별 분포·breaking 영향 비율·긴급 대응 필요 건수 등 KPI를 확인한다.

### 시나리오 3 — 제출 전 사전 검증(검증 전용 흐름)

릴리스 담당자가 실제 제출(저장) 전에 동일한 입력값으로 검증 전용 라우트를 호출해, 저장 없이 유효성 결과(통과 여부 및 오류 상세)만 미리 확인한다. 이 호출은 요약 보드나 KPI에 어떠한 흔적도 남기지 않는다.

### 시나리오 4 — 검증 실패 에러 흐름

릴리스 담당자가 제목/변경 항목을 비워두거나, 정의되지 않은 중요도·사용자 영향도 값을 입력하는 등 잘못된 요청을 하면, 서버는 카드를 생성하지 않고 어떤 필드가 왜 실패했는지 명확히 알려주는 오류를 반환한다.

## 3. 데이터 모델

### 3.1 ReleaseNoteSummaryCard 레코드

```json
{
  "id": "string (uuid v4)",
  "title": "string, 공백 제거 후 1~200자, 필수",
  "changes": "string[], 최소 1개 ~ 최대 50개, 각 항목 공백 제거 후 1~300자, 필수",
  "importance": "string, enum: low | medium | high | critical, 필수",
  "userImpact": "string, enum: none | minor | major | breaking, 필수",
  "createdAt": "string, ISO 8601 UTC 타임스탬프, 서버 생성, 불변"
}
```

| 필드 | 타입 | 필수 | 규칙 |
|---|---|---|---|
| `id` | string | Y | UUID v4, 서버 생성, 불변 |
| `title` | string | Y | 공백 제거(trim) 후 1~200자. 빈 문자열/공백뿐이면 거부 |
| `changes` | string[] | Y | 최소 1개 이상. 각 항목 trim 후 1~300자. 빈 배열이거나 빈 문자열 항목이 하나라도 있으면 거부 |
| `importance` | enum | Y | `low`, `medium`, `high`, `critical` 중 하나. 그 외 값(대소문자 상이 포함)은 거부 |
| `userImpact` | enum | Y | `none`, `minor`, `major`, `breaking` 중 하나. 그 외 값은 거부 |
| `createdAt` | ISO8601 | Y | 생성 시 서버가 1회 설정, 이후 변경 불가 |

- 수정/삭제 API는 본 범위에 없다. 생성된 카드는 조회 전용이다(§10 Non-goals).

### 3.2 중요도(`importance`) vs 사용자 영향도(`userImpact`) 구분 규칙

두 필드는 서로 **독립적인 축**이며, 하나가 다른 하나를 함의하지 않는다.

| 구분 | `importance` (중요도) | `userImpact` (사용자 영향도) |
|---|---|---|
| 관점 | 릴리스 담당/운영 조직 내부의 **우선순위·대응 리소스 배정 기준** | 최종 사용자가 **체감하는 변경 범위·커뮤니케이션 강도** 기준 |
| 값 | `low`/`medium`/`high`/`critical` | `none`/`minor`/`major`/`breaking` |
| 사용처 예 | 배포 순서, 온콜 대응 우선순위, 내부 알림 강조 | 공지 문구 강조 레벨, 별도 마이그레이션 안내 필요 여부 |
| 독립성 예시 1 | `importance=critical`, `userImpact=none` — 내부 인프라/보안 패치로 운영상 시급하지만 사용자는 아무 변화도 못 느낌 |
| 독립성 예시 2 | `importance=low`, `userImpact=breaking` — 운영 우선순위는 낮지만 소수 사용자에게는 API 호환성이 깨지는 변경 |

- 두 필드는 각각 독립적으로 검증되며, 한쪽 값이 다른 쪽 값의 허용 범위를 제한하지 않는다(예: `userImpact=breaking`이라고 `importance`가 반드시 `high` 이상일 필요는 없음).
- 요약 카드 UI/배지 표현(시각적 강조 방식)은 BF-1086(designer) 재량이며, 본 문서는 값의 의미·구분 규칙만 정의한다.

## 4. 검증 규칙 — 빈 입력 / 잘못된 중요도

| 대상 필드 | 실패 조건 | 처리 |
|---|---|---|
| `title` | 누락, 빈 문자열, 공백만 존재, 200자 초과 | 거부 — 오류 코드에 `title` 명시 |
| `changes` | 누락, 빈 배열, 51개 이상, 항목 중 하나라도 빈 문자열/공백뿐/300자 초과 | 거부 — 오류 코드에 `changes` 및 실패한 인덱스 명시 |
| `importance` | 누락, 정의된 4개 enum 값 이외 (대소문자 상이·오탈자 포함) | 거부 — 오류 코드에 `importance`와 허용값 목록 명시 |
| `userImpact` | 누락, 정의된 4개 enum 값 이외 | 거부 — 오류 코드에 `userImpact`와 허용값 목록 명시 |
| 요청 본문 자체가 malformed JSON | 파싱 불가 | 거부 — 필드 단위 오류가 아닌 요청 레벨 오류로 명확히 구분 |

- 여러 필드가 동시에 실패할 수 있으며, 이 경우 **모든 실패 필드를 한 번에** 오류 목록에 담아 반환한다(첫 번째 오류만 보고 나머지를 감추지 않는다).
- 하나라도 실패하면 카드는 생성/저장되지 않는다(부분 저장 금지).

## 5. API 스펙 (초안 — 요청/응답 계약)

구현 프레임워크는 규정하지 않으며, 아래는 개념적 요청/응답 계약이다.

### 5.1 요약 카드 생성

- `POST /release-notes`
- Request body: `{ "title": "string", "changes": ["string", ...], "importance": "low|medium|high|critical", "userImpact": "none|minor|major|breaking" }`
- 성공 응답 (201): 생성된 ReleaseNoteSummaryCard 레코드 전체(§3.1)
- 실패 응답 (400): §4 검증 규칙 위반 시 — 실패 필드 목록과 사유를 포함한 오류 상세

### 5.2 검증 전용 라우트 (side-effect 없음)

- `POST /release-notes/validate`
- Request body: 5.1과 **동일한 스키마**
- §4의 검증 규칙을 그대로 적용하되, **결과와 무관하게 항상 200**으로 응답하고 저장/ID 발급 등 어떠한 부작용도 발생시키지 않는다.
- 성공 응답 (200): `{ "valid": true }`
- 실패 시에도 응답 (200): `{ "valid": false, "errors": [ { "field": "importance", "reason": "..." }, ... ] }`
- 이 라우트 호출은 §6 KPI, §5.3/5.4 목록·KPI 조회 결과에 **어떤 영향도 주지 않는다**(§9 검증 전용 라우트 범위 참고).

### 5.3 요약 보드 목록 조회

- `GET /release-notes?importance=<enum|all>&userImpact=<enum|all>`
- 쿼리 파라미터 생략 또는 `all`이면 필터 미적용
- 두 필터는 AND 결합 (예: `importance=critical&userImpact=breaking` → 둘 다 만족하는 카드만)
- 성공 응답 (200): 생성 순서(오래된 순) 배열. 결과 0건이면 빈 배열 `[]` (에러 아님)

### 5.4 KPI 조회

- `GET /release-notes/kpi`
- 성공 응답 (200): §6에 정의된 KPI 필드를 담은 객체 (현재 전체 데이터 기준, 필터 미적용)

## 6. KPI 측정 항목

모든 KPI는 조회 시점의 전체 요약 카드 데이터(필터 미적용, §5.2 검증 전용 호출은 제외) 기준으로 매번 재계산한다.

| KPI | 정의 | 계산 방법 |
|---|---|---|
| 총 카드 건수 | 현재 저장된 전체 요약 카드 수 | `cards.length` |
| 중요도별 분포 | 중요도별 건수 및 비율 | `cards`를 `importance`로 그룹핑한 개수 / 비율 = `count / 전체 * 100`(소수점 1자리). 전체 0건이면 각 비율 `0%` |
| 사용자 영향도별 분포 | 영향도별 건수 및 비율 | `cards`를 `userImpact`로 그룹핑, 계산 방식은 위와 동일 |
| Breaking 영향 비율 | 사용자 영향이 큰 변경 경보 지표 | `userImpact === 'breaking'` 건수 / 전체 * 100 (%, 소수점 1자리). 전체 0건이면 `0%` |
| 긴급 대응 필요 건수 | 우선 대응 신호 | `importance`가 `high` 또는 `critical`인 건수 |
| 최근 등록 추이 | 최근 유입 속도 | 조회 시각 기준 최근 24시간 이내 `createdAt`을 가진 카드 건수. 24시간은 초기 기본값이며 조정 가능(구현 재량) |

- 0으로 나누기 방지: 전체 건수가 0이면 모든 비율성 KPI는 `0%`로 표시하고, 절대 오류를 던지지 않는다.
- §5.2 검증 전용 라우트 호출은 카드를 생성하지 않으므로 KPI 계산에 전혀 반영되지 않는다.

## 7. Edge Case / 에러 케이스 요약

| 케이스 | 처리 방침 |
|---|---|
| `title`이 빈 문자열이거나 공백만 포함 | 생성 거부 (400) |
| `title`이 200자 초과 | 생성 거부 (400) |
| `changes`가 빈 배열이거나 항목 중 빈 문자열/공백뿐 포함 | 생성 거부 (400), 실패 인덱스 명시 |
| `importance`가 정의된 enum 값이 아님 | 생성 거부 (400), 허용값 목록 안내 |
| `userImpact`가 정의된 enum 값이 아님 | 생성 거부 (400), 허용값 목록 안내 |
| 요청 본문이 malformed JSON | 생성 거부, 요청 레벨 오류로 명확히 구분해 400 |
| 검증 전용 라우트(`/validate`)에 위 실패 조건 입력 | 200과 함께 `{ valid: false, errors: [...] }` 반환(§5.2), 400 아님 |
| 존재하지 않는/의미 없는 필터 값으로 목록 조회 | 서버 오류 아님 — 빈 배열 `[]` 반환 (empty state) |
| 필터 결과 0건 | empty state 표시, 오류 아님 |
| 카드 0건 상태에서 KPI 조회 | 총 건수 0, 모든 비율 `0%`, 최근 등록 추이 0건으로 정상 응답 |

## 8. Acceptance Criteria (Given/When/Then)

### AC-1. 정상 생성 (시나리오 1)

- Given 릴리스 담당자가 유효한 `title`/`changes`/`importance`/`userImpact`를 입력했을 때
- When 생성을 요청하면(`POST /release-notes`)
- Then 서버가 고유 `id`와 `createdAt`을 부여해 요약 카드를 저장하고, 생성된 카드를 201로 반환한다.

### AC-2. 빈 입력·잘못된 중요도 검증 실패 (시나리오 4)

- Given `title`/`changes`가 비어있거나, `importance`/`userImpact`가 정의된 enum 값이 아닐 때
- When 생성을 요청하면
- Then 카드가 저장되지 않고, 실패한 모든 필드를 포함한 400 오류가 반환된다(§4).

### AC-3. 검증 전용 라우트의 무부작용 보장 (시나리오 3)

- Given 임의의 유효/무효 입력으로 `POST /release-notes/validate`를 호출했을 때
- When 요청이 처리되면
- Then 결과(valid/invalid)와 무관하게 항상 200으로 응답하고, 카드가 저장되지 않으며 §6 KPI·§5.3 목록 조회 결과에 어떠한 변화도 발생하지 않는다.

### AC-4. 중요도/영향도 독립성 (데이터 모델)

- Given `importance`와 `userImpact`에 서로 다른 조합(예: `critical`/`none`, `low`/`breaking`)이 입력됐을 때
- When 카드가 생성되면
- Then 두 필드는 서로의 값을 제한하지 않고 각각 독립적으로 저장·검증·집계된다(§3.2).

### AC-5. 요약 보드 필터 조회 및 KPI 집계 (시나리오 2)

- Given 중요도·사용자 영향도 필터가 각각 `all`이 아닌 값으로 설정됐을 때
- When 목록을 조회하면(`GET /release-notes`)
- Then 두 조건을 AND로 만족하는 카드만 생성 순서대로 반환되고(없으면 빈 배열), KPI 조회 시(`GET /release-notes/kpi`) §6에 정의된 총 건수·분포·breaking 비율·긴급 대응 건수·최근 등록 추이가 정확히 계산되며 0건일 때 모든 비율은 `0%`로 반환된다(0-division 방지).

### AC-6. 기존 라우트/데이터 무변경 (마이그레이션 전략)

- Given 본 기능(§5 라우트 4종)이 기존 시스템에 추가될 때
- When 배포·마이그레이션이 수행되면
- Then 기존 라우트의 동작·응답 스키마와 기존에 저장된 데이터는 전혀 변경되지 않으며, 본 기능은 별도 데이터 마이그레이션 없이 완전히 독립적으로 추가되는 신규 기능임이 보장된다(§10).

---

## 9. 검증 전용 라우트 범위 (Scope)

- **포함**: `POST /release-notes/validate` 1개 라우트만 해당. §3.1 스키마에 대한 §4 검증 규칙 적용, 저장/ID 발급/`createdAt` 부여 없음, 응답은 항상 200 + `{ valid, errors? }`.
- **미포함**: 목록 필터링 로직 검증, KPI 계산 로직 검증, 인증/권한 검사, rate limiting — 본 라우트는 순수 입력 스키마 유효성만 다룬다.
- **부작용 없음(no side effects) 보장**: 이 라우트를 호출한 이력 자체를 저장하거나 로깅해 사용자에게 노출하는 기능은 본 범위에 없다(내부 운영 로깅 여부는 구현 재량, 단 사용자 대면 데이터에는 영향 없어야 함).
- **호출 빈도 제한 없음**: 동일 입력을 반복 검증해도 결과는 항상 동일(idempotent, side-effect free)해야 한다.

## 10. 마이그레이션 전략 — 독립 기능, 무변경 원칙

- 본 기능은 `release-notes`라는 새 리소스/네임스페이스에서만 동작하며, 기존 라우트(예: 다른 모듈의 API)나 기존 저장 데이터 스키마를 **전혀 변경하지 않는다**.
- 기존 데이터를 신규 스키마로 옮기는 백필/이관 작업이 필요 없다 — 처음부터 빈 상태(`총 카드 0건`)로 시작하는 신규 기능이다.
- 롤백 시나리오: 본 기능을 제거하더라도 기존 기능·데이터에는 영향이 없어야 한다(완전 분리).
- 저장 방식(인메모리/파일/DB)은 본 문서에서 규정하지 않으며, BF-1087(developer)이 저장소 실제 규약(§ 상단 스택 불일치 메모)에 맞춰 결정한다. 단, 어떤 저장 방식을 택하더라도 §5~§9의 요청/응답 계약과 §4 검증 규칙은 동일하게 지켜져야 한다.

## 12. Non-goals (본 범위 밖)

- 요약 카드 수정/삭제, 상태 워크플로(승인/게시 등 라이프사이클 전이) — 본 서비스는 생성·검증·조회·집계만 다룬다.
- 인증/권한, 다중 사용자 동시 편집 충돌 해결.
- 대용량 트래픽 대비 페이지네이션·아카이빙·축출(eviction) 정책 — 필요 시 별도 후속 Task로 다룬다.
- 검증 전용 라우트(§9)를 통한 실제 게시/알림 발송 — 순수 입력 유효성 확인 전용이다.
- 시안(비주얼 디자인)은 BF-1086(designer) 담당, 실제 구현은 BF-1087(developer) 담당.

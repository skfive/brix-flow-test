# 고객 피드백 펄스 서비스 기획 명세 (BF-1079)

- 대상 모듈: `feedback-pulse` (접수 API + 펄스 보드 조회/KPI)
- 관련 형제 Task: BF-1080(designer, 시안), BF-1081(developer, 구현), BF-1083(tester, 검증)
- 본 문서는 **기획 명세만** 다룬다. 코드/시안은 포함하지 않는다.
- **저장 전제(중요)**: 모든 데이터는 **프로세스 인메모리**에만 보관한다. 영속 저장소(파일/DB)는 사용하지 않으며, **서버 프로세스가 재시작되면 저장된 모든 피드백·KPI 누적치는 초기화(소실)된다.** 이는 버그가 아니라 본 서비스의 설계 전제이며, 운영자에게 사전 고지되어야 한다.
- **기술 스택 참고(가정 명시)**: 요청 설명에는 `nodejs-backend` 마커가 있으나, 저장소 관측 규약(`base_sha` 기준)은 `vanilla-static`이며 두 값이 불일치한다. 본 문서는 구현 프레임워크를 규정하지 않고 **동작 계약(요청/응답, 데이터 모델, 상태 전제)만** 정의한다. 실제 구현 형태(경량 인메모리 서비스 모듈 vs 별도 서버 프로세스)는 BF-1081(developer)이 저장소 실제 규약에 맞춰 결정한다.

---

## 1. 개요

고객이 실시간으로 남기는 피드백(감정·긴급도·의견)을 접수하고, 운영자가 "펄스 보드"에서 현재 접수 현황과 KPI를 한눈에 확인할 수 있는 서비스. 별도 DB 없이 프로세스 인메모리에만 데이터를 보관하는 경량 서비스로, 재시작 시 데이터가 초기화되는 것을 전제로 한다.

---

## 2. 사용자 시나리오

### 시나리오 1 — 피드백 접수

사용자가 감정(긍정/중립/부정), 긴급도(낮음/보통/높음/긴급), 의견(자유 텍스트)을 입력해 피드백을 접수한다. 접수 즉시 서버는 고유 ID와 접수 시각을 부여해 인메모리 목록에 추가한다.

### 시나리오 2 — 펄스 보드 조회 및 KPI 확인

운영자가 펄스 보드에서 현재까지 접수된 피드백 목록을 감정/긴급도로 필터링해 보고, 화면 상단에서 총 접수 건수·감정별 분포·긴급도별 분포·부정 감정 비율·긴급 대응 필요 건수·최근 접수 추이(펄스율) 등 KPI를 확인한다.

### 시나리오 3 — 접수/조회 에러 흐름

사용자가 의견을 비워두거나 감정/긴급도에 정의되지 않은 값을 보내는 등 잘못된 요청을 하면, 서버는 데이터를 저장하지 않고 명확한 오류를 반환한다. 또한 서버 프로세스가 재시작된 직후에는 이전 데이터가 전혀 없는 빈 상태(KPI 전부 0)로 정상 동작해야 한다.

## 3. 데이터 모델

### 3.1 FeedbackPulse 레코드

```json
{
  "id": "string (uuid v4)",
  "sentiment": "string, enum: positive | neutral | negative, 필수",
  "urgency": "string, enum: low | medium | high | critical, 필수",
  "opinion": "string, 공백 제거 후 1~1000자, 필수",
  "submittedAt": "string, ISO 8601 UTC 타임스탬프, 서버 생성, 불변"
}
```

| 필드 | 타입 | 필수 | 규칙 |
|---|---|---|---|
| `id` | string | Y | UUID v4, 서버 생성, 불변 |
| `sentiment` | enum | Y | `positive`, `neutral`, `negative` 중 하나. 그 외 값은 접수 거부 |
| `urgency` | enum | Y | `low`, `medium`, `high`, `critical` 중 하나. 그 외 값은 접수 거부 |
| `opinion` | string | Y | 공백 제거 후 1~1000자. 빈 문자열이거나 1000자 초과 시 접수 거부 |
| `submittedAt` | ISO8601 | Y | 접수 시 서버가 1회 설정, 이후 변경 불가 |

- 상태 전이(워크플로) 개념은 본 서비스 범위에 없다. 접수된 레코드는 수정·삭제 없이 조회 전용으로만 사용된다(§12 Non-goals).

### 3.2 인메모리 저장 구조 (개념 모델)

```json
{
  "items": [ /* FeedbackPulse 레코드 배열, 접수 순서(FIFO) 유지 */ ]
}
```

- 프로세스 시작 시 `items: []`로 초기화된다.
- 접수는 항상 배열 끝에 append(FIFO), 별도 정렬 로직 없이 접수 순서가 곧 저장 순서다.
- 디스크/DB/localStorage 등 어떤 영속 저장소에도 기록하지 않는다.

---

## 4. 저장 정책 — 인메모리 전제 및 초기화

- **단일 프로세스 인메모리 저장**만 사용한다. 프로세스가 재시작(배포, 크래시, 재기동 등)되면 `items`는 빈 배열로 완전히 초기화되고 이전 데이터는 복구할 수 없다.
- 이 초기화는 별도의 "복구/백업" 로직을 필요로 하지 않는다 (BF-1072의 localStorage 손상 복구와 달리, 애초에 영속 저장소가 없으므로 손상 개념 자체가 없다).
- 운영자 대상 안내: 펄스 보드 화면 또는 운영 문서에 "이 데이터는 서버 재시작 시 초기화됩니다"라는 전제를 노출해야 한다(문구 형태는 designer/developer 재량).
- 동시 접수 시 순서 보장: 단일 프로세스 내에서 접수 요청은 도착한 순서대로 `items`에 append되어야 하며, 두 요청이 거의 동시에 도착해도 각 레코드는 서로 다른 `id`를 가져야 한다(ID 충돌 금지).

## 5. API 스펙 (초안 — 요청/응답 계약)

구현 프레임워크는 규정하지 않으며, 아래는 개념적 요청/응답 계약이다.

### 5.1 피드백 접수

- `POST /feedback-pulse`
- Request body: `{ "sentiment": "positive|neutral|negative", "urgency": "low|medium|high|critical", "opinion": "string" }`
- 성공 응답 (201): 생성된 FeedbackPulse 레코드 전체(§3.1)
- 실패 응답 (400): 유효성 오류 — 필수값 누락, enum 값 불일치, opinion 길이 초과/공백, malformed JSON 등 (§7)

### 5.2 펄스 보드 목록 조회

- `GET /feedback-pulse?sentiment=<enum|all>&urgency=<enum|all>`
- 쿼리 파라미터 생략 또는 `all`이면 필터 미적용
- 두 필터는 AND 결합 (예: `sentiment=negative&urgency=high` → 둘 다 만족하는 레코드만)
- 성공 응답 (200): 접수 순서(오래된 순) 배열. 결과 0건이면 빈 배열 `[]` (에러 아님)

### 5.3 KPI 조회

- `GET /feedback-pulse/kpi`
- 성공 응답 (200): §6에 정의된 KPI 필드를 담은 객체 (현재 인메모리 전체 데이터 기준, 필터 미적용)

---

## 6. KPI 측정 항목

모든 KPI는 조회 시점의 인메모리 전체 데이터(필터 미적용) 기준으로 매번 재계산한다.

| KPI | 정의 | 계산 방법 |
|---|---|---|
| 총 접수 건수 | 현재 인메모리에 있는 전체 레코드 수 | `items.length` |
| 감정별 분포 | 감정별 건수 및 비율 | `items`를 `sentiment`로 그룹핑한 개수 / 비율 = `count / 전체 * 100`(소수점 1자리). 전체 0건이면 각 비율 `0%` |
| 긴급도별 분포 | 긴급도별 건수 및 비율 | `items`를 `urgency`로 그룹핑, 계산 방식은 위와 동일 |
| 부정 감정 비율 | 부정 피드백 경보 지표 | `sentiment === 'negative'` 건수 / 전체 * 100 (%, 소수점 1자리). 전체 0건이면 `0%` |
| 긴급 대응 필요 건수 | 우선 대응 신호 | `urgency`가 `high` 또는 `critical`인 건수 |
| 최근 접수 추이(펄스율) | 실시간 유입 속도 | 조회 시각 기준 최근 5분 이내 `submittedAt`을 가진 레코드 건수. 5분은 초기 기본값이며 조정 가능(구현 재량) |

- 0으로 나누기 방지: 전체 건수가 0이면 모든 비율성 KPI는 `0%`로 표시하고, 절대 오류를 던지지 않는다.

## 7. Edge Case / 에러 케이스 요약

| 케이스 | 처리 방침 |
|---|---|
| `opinion`이 빈 문자열이거나 공백만 포함 | 접수 거부 (400) |
| `opinion`이 1000자 초과 | 접수 거부 (400) |
| `sentiment`가 정의된 enum 값이 아님 | 접수 거부 (400) |
| `urgency`가 정의된 enum 값이 아님 | 접수 거부 (400) |
| 요청 본문이 malformed JSON | 접수 거부, 명확한 오류 메시지와 함께 400 |
| 존재하지 않는/의미 없는 필터 값으로 조회 | 서버 오류 아님 — 빈 배열 `[]` 반환 (empty state) |
| 필터 결과 0건 | empty state 표시, 오류 아님 |
| 서버 프로세스 재시작 직후 조회 | 빈 목록·KPI 전부 0(비율은 `0%`, 펄스율 0건)으로 정상 응답 (§4) |
| 동시 다발 접수(거의 동시 도착) | 각 레코드가 서로 다른 고유 `id`를 가지며 접수 순서(FIFO)가 유지됨, 유실/중복 없음 |
| 장시간 운영으로 인한 무제한 메모리 증가 | 본 범위에서는 별도 축출(eviction)·영속화 정책을 두지 않음 — 알려진 제약으로 명시 (§12 Non-goals) |

## 8. Acceptance Criteria (Given/When/Then)

### AC-1. 정상 접수 (시나리오 동작)

- Given 사용자가 유효한 `sentiment`/`urgency`/`opinion`을 입력했을 때
- When 접수를 요청하면(`POST /feedback-pulse`)
- Then 서버가 고유 `id`와 `submittedAt`을 부여해 인메모리 목록에 append하고, 생성된 레코드를 201로 반환한다.

### AC-2. 접수 유효성 실패 (에러 흐름)

- Given `opinion`이 공백뿐이거나, `sentiment`/`urgency`가 enum 값이 아니거나, 요청 본문이 malformed JSON일 때
- When 접수를 요청하면
- Then 인메모리 목록에 반영되지 않고 400과 함께 구체적인 오류가 반환된다.

### AC-3. 펄스 보드 필터 조회 (시나리오 동작)

- Given 감정·긴급도 필터가 각각 `all`이 아닌 값으로 설정됐을 때
- When 목록을 조회하면(`GET /feedback-pulse`)
- Then 두 조건을 AND로 만족하는 레코드만 접수 순서대로 반환되고, 없으면 빈 배열이 반환된다(에러 아님).

### AC-4. KPI 집계 계산 (KPI 코드)

- Given 인메모리에 0건 이상의 데이터가 있을 때
- When KPI를 조회하면(`GET /feedback-pulse/kpi`)
- Then §6에 정의된 총 건수·감정별/긴급도별 분포·부정 감정 비율·긴급 대응 필요 건수·펄스율이 정확히 계산되어 반환되고, 전체 건수가 0이면 모든 비율성 KPI는 `0%`로 반환된다(0-division 방지).

### AC-5. 프로세스 재시작 시 인메모리 초기화 (롤백)

- Given 서버 프로세스가 재시작되어 이전 인메모리 데이터가 전부 소실된 상태일 때
- When 목록 또는 KPI를 조회하면
- Then 목록은 빈 배열, KPI는 총 건수 0·비율 `0%`·펄스율 0건으로 정상 응답하며 서버는 에러를 발생시키지 않는다.

### AC-6. 동시 접수 순서 보장 (시나리오 동작)

- Given 두 건 이상의 접수 요청이 거의 동시에 도착했을 때
- When 각 요청이 처리되면
- Then 각 레코드는 서로 다른 고유 `id`를 가지며, 접수된 순서대로 `items`에 append되어 유실이나 덮어쓰기가 발생하지 않는다.

## 10. Non-goals (본 범위 밖)

- 영속 저장소(파일/DB/localStorage) 연동 — 인메모리 전용이며 재시작 시 초기화가 설계 전제다(§4).
- 레코드 수정/삭제, 상태 워크플로(예: BF-1072의 open/planned/done 같은 상태 전이) — 본 서비스는 접수·조회·집계만 다룬다.
- 대용량 트래픽 대비 메모리 축출(eviction)·페이지네이션·영속 아카이빙 — 알려진 제약으로만 명시(§7).
- 인증/권한, 다중 서버 인스턴스 간 데이터 동기화(클러스터링).
- 시안(비주얼 디자인)은 BF-1080(designer) 담당, 실제 구현은 BF-1081(developer) 담당.

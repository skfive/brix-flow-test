# 스킬 패키지 운영 5단계 체크리스트 (BF-1127)

> 본 문서는 `prd-writer` Skill 지침(Problem Statement → Proposed Solution → Requirements →
> Implementation → Risks and Mitigations)을 짧은 운영 체크리스트 형식에 맞춰 적용한 기획 문서다.

## 1. Problem Statement

- **현재 상황**: 운영자가 스킬 패키지를 설치·배정한 뒤, 해당 스킬이 실제로 정상 동작하는지
  확인하는 표준 절차가 없다. 페르소나마다 확인 방식이 달라 누락이 발생한다.
- **User pain point**: 스킬이 `Unknown skill` 오류로 실패하거나, 지침이 실제 산출물에
  반영되지 않은 채로 Run이 종료되어도 운영자가 사후에야 인지한다.
- **Business impact**: 잘못 배정된 스킬로 인한 Run 실패는 재작업 비용과 Jira/PR 노이즈를
  유발하고, 운영자의 스킬 패키지 신뢰도를 낮춘다.

## 2. Proposed Solution

### 개요
스킬 패키지 등록 직후 운영자가 순서대로 확인하는 5단계 체크리스트를 정의한다. 각 단계는
`목적 → 확인 항목 → 통과 기준(Pass Criteria)` 3요소로 구성해, PRD의 "acceptance criteria
포함" 원칙을 그대로 따른다.

### 5단계 운영 체크리스트

#### 1단계 — 패키지 등록 확인
- **목적**: 스킬 패키지가 설치 레지스트리에 정상 등록되었는지 확인한다.
- **확인 항목**:
  - 스킬 패키지 메타데이터(이름, 설명, 버전)가 설치 목록에 노출되는가
  - `search_installed_skills` 등 조회 경로로 해당 스킬을 검색할 수 있는가
- **통과 기준**: 검색 결과에 스킬 이름·설명이 정확히 표시되고 중복/누락이 없다.

#### 2단계 — Run 배정(assignment) 확인
- **목적**: 대상 페르소나의 Run에 스킬이 `assigned_skills` 또는
  `required_for_task_skills`로 정확히 주입되었는지 확인한다.
- **확인 항목**:
  - Run capability manifest에 스킬의 `invocation_ref`(exact path)가 명시되어 있는가
  - 필수/선택 배정 여부가 task 요구사항과 일치하는가
- **통과 기준**: manifest의 invocation_ref가 실제 설치 경로와 일치하고, 필수 스킬 목록에
  누락된 항목이 없다.

#### 3단계 — 최초 호출(activation) 성공 확인
- **목적**: 페르소나가 실행 초기에 Skill 도구로 해당 스킬을 성공적으로 호출하는지 확인한다.
- **확인 항목**:
  - Skill 도구 호출 결과가 `Unknown skill` 등 실패 응답이 아닌가
  - 호출 시점이 일반 작업 시작 이전(초기)인가
- **통과 기준**: 최초 호출이 실패 없이 스킬 본문(지침)을 반환하고, 호출 로그가 작업 기록에
  남는다.

#### 4단계 — 지침의 산출물 반영 확인
- **목적**: 스킬이 제시한 구조·항목이 실제 산출물 문서/코드에 반영되었는지 확인한다.
- **확인 항목**:
  - 스킬 지침의 핵심 섹션(예: 문제 정의, 요구사항, 리스크 등)이 산출물에서 식별 가능한가
  - 지침을 형식적으로 언급만 하지 않고 내용에 실질 반영했는가
- **통과 기준**: 산출물을 스킬 지침 없이 읽어도 지침이 요구한 구조적 요소를 모두 확인할 수
  있다.

#### 5단계 — 범위·부작용 최종 점검
- **목적**: 스킬 사용이 담당 파일 범위(owned_paths)와 수용 기준을 벗어나지 않았는지
  최종 확인한다.
- **확인 항목**:
  - 변경된 파일이 계약된 owned_paths와 정확히 일치하는가
  - 스킬 호출로 인한 부가 side-effect(불필요한 외부 호출, 범위 밖 수정 등)가 없는가
- **통과 기준**: `git status`/`git diff` 상 owned_paths 외 변경이 없고, 스킬 호출 기록이
  작업 범위 안에서만 발생했다.

## 3. Requirements

- **Functional**: 5단계는 순차적으로 수행되며, 앞 단계가 실패하면 다음 단계로 진행하지
  않고 즉시 실패로 보고한다.
- **Technical**: 각 단계의 확인 항목은 Run capability manifest, Skill 도구 호출 결과,
  `git status`/`git diff` 등 이미 존재하는 산출물로 검증 가능해야 하며 별도 인프라를
  요구하지 않는다.
- **Design**: 체크리스트는 운영자가 화면에서 한눈에 훑을 수 있도록 단계당 3줄 이내
  (목적/확인 항목/통과 기준)로 유지한다.

## 4. Implementation

- **Dependencies**: 스킬 설치·배정 시스템(브릭스 컨트롤), Run capability manifest 노출
  기능이 선행되어 있어야 한다.
- **Timeline**: 별도 개발 없이 즉시 운영 적용 가능한 체크리스트 문서이므로 타임라인은
  운영자 교육/공지 1회로 충분하다.
- **Resources**: 운영자 1인이 등록 직후 5~10분 내 점검 가능한 분량으로 설계했다.

## 5. Risks and Mitigations

- **Risk**: 운영자가 3단계(최초 호출 성공)를 생략하고 4단계로 건너뛸 수 있다.
  **Mitigation**: 체크리스트 순서를 강제하고, 3단계 실패 시 이후 단계를 "통과 불가"로
  명시한다.
- **Risk**: 지침 반영 여부(4단계) 판단이 주관적일 수 있다.
  **Mitigation**: 통과 기준을 "스킬 지침 없이 읽어도 구조적 요소 확인 가능"으로 객관화한다.
- **Risk**: owned_paths 밖 변경이 뒤늦게 발견될 수 있다.
  **Mitigation**: 5단계에서 `git status`/`git diff` 확인을 필수 항목으로 고정한다.

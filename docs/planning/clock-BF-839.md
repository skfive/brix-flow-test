## 0. 문서 성격 및 전제 (필독 — 재해석 금지)

**가정 1 (모호하여 명시함) — `tech-stack:nextjs` 태그 불일치:** Epic 메타데이터에 `<!-- bf:tech-stack:nextjs -->` 태그가 붙어 있으나, 본 저장소(`brix-flow-test`)는 Next.js 프로젝트가 아니다. `package.json`(`name: notepad-spa`)에는 Next 관련 의존성이 0건이며, `timer/`·`stopwatch/`·`pomodoro/`·`rps/`·`incident-command/` 등 기존 모든 신규 모듈이 **바닐라 HTML/CSS/JS + `file://` 직접 실행 호환** 패턴(`vanilla-static`)으로 구현되어 있다. 저장소 전체를 Next.js로 전환하는 것은 본 task(문서 1건 산출) 범위를 명백히 벗어나는 아키텍처 변경이므로, Simplicity First 원칙에 따라 본 문서는 기존 컨벤션을 그대로 따르는 `clock/` vanilla-static 모듈로 스펙을 작성한다. 실제로 Next.js 전환을 의도한 것이라면 별도 확인이 필요하다(§12-1).

**가정 2 — `/demo/clock` 경로 표기:** Epic 설명은 "`/demo/clock` 시계 SPA"라고 표기하지만, 기존 신규 프론트엔드 모듈은 전부 저장소 루트의 독립 디렉터리(`timer/`, `stopwatch/`, `pomodoro/` 등)에 위치하며 `/demo/` prefix 라우트는 존재하지 않는다(유일한 선례인 BF-833/834 tetris 문서의 `/demo/tetris`는 **신규 백엔드 API 네임스페이스**였고 프론트엔드는 그대로 `tetris/`였다). 본 문서는 "`/demo/clock`"을 향후 구현될 `clock/index.html` SPA를 가리키는 설명적 표현으로 해석하고, 신규 모듈 경로는 기존 컨벤션과 동일하게 저장소 루트 `clock/`로 정의한다.

**가정 3 — 완전 신규(forward) 모듈:** `clock/` 디렉터리·코드·테스트는 현재 저장소에 0건 존재한다(사전 확인 완료). 본 문서는 `rps-BF-789.md`와 동일한 성격 — 기존 구현을 역공식화하는 것이 아니라 **아직 존재하지 않는 모듈의 최초 스펙**을 정의하는 forward 설계 문서다.

**가정 4 — 파일 소유권:** 본 planner task(BF-840)의 담당 파일은 `docs/planning/clock-BF-839.md` 1개뿐이다. `clock/*` 코드, `docs/design/*` 문서는 후속 designer/dev 담당 영역이며 본 task 에서 생성·수정하지 않는다.

**가정 5 — "완료 기준 5항목"의 해석:** 본 task 의 세 번째 수용 기준은 "완료 기준 5항목이 검증 가능한 형태로 매핑된다"를 요구하나, 별도의 명시적 "5개 항목" 목록이 Jira 상 첨부되어 있지 않다(본 세션에는 Jira 조회 도구가 연결되어 있지 않아 BF-839 Epic 원문을 직접 조회할 수 없었다). 본 task 설명 문장 "표시 규칙(로컬 날짜, HH:MM:SS)과 **시작·1초 갱신·12/24시간 전환·정지/재개** 시나리오"를 근거로, 아래 5개 항목으로 분해하여 매핑한다(§8):

## 1. 개요

### 1.1 목적

브라우저에 로컬 시스템 시각을 실시간으로 표시하는 최소 SPA. 외부 API·서버 시각 동기화 없이 오직 클라이언트의 `Date` 객체만으로 동작하며, 1초마다 자동 갱신되고 사용자가 정지/재개, 12/24시간 표시 형식을 전환할 수 있다.

### 1.2 적용 범위

| 항목 | 내용 |
|---|---|
| 신규 경로 | `clock/`(`index.html` / `styles.css` / `clock.js` / `main.js`) — 저장소 루트, 기존 모듈과 동일 패턴 |
| 기존 코드 영향 | 없음 — 완전 독립 신규 모듈 |
| 데이터 원천 | 클라이언트 `Date` 객체(로컬 시스템 시각)뿐 — 서버/외부 API·NTP 동기화 없음 |
| 외부 라이브러리 | 없음 — `file://` 프로토콜 직접 열기 가능 |
| 영속 저장 | `localStorage`에 12/24시간 표시 형식 설정만 저장(선택 사항, §5) — 정지/재개 상태 자체는 저장하지 않음(§9) |

### 1.3 전제 조건

- 브라우저 환경(Chrome/Edge/Firefox 최신 버전) 또는 Node.js(`node --test`)로 순수 함수(포맷팅 로직) 단위 테스트
- `clock/` 디렉터리는 아직 존재하지 않으며 본 task 이후 별도 designer/dev task 에서 신규 생성됨
- 포맷팅 순수 함수(`formatDate`, `formatTime` 등)는 UMD 패턴으로 작성되어 Node(CommonJS) + 브라우저(`globalThis`) 양쪽에서 로드 가능해야 함(기존 `rps/logic.js`, `incident-command/fixtures.js` 등과 동일 컨벤션)

## 3. 표시 규칙 — 로컬 날짜 · HH:MM:SS

### 3.1 날짜 표시 형식

브라우저 로케일에 의존하는 `toLocaleDateString()` 은 실행 환경(OS 언어/지역 설정)에 따라 출력 문자열이 달라져 **결정론적 테스트가 불가능**하므로 채택하지 않는다. 대신 아래 고정 포맷을 자체 계산으로 생성한다:

```
YYYY-MM-DD (요일)
예: 2026-07-16 (목)
```

| 요소 | 규칙 |
|---|---|
| `YYYY` | `date.getFullYear()`, 4자리 |
| `MM` | `date.getMonth() + 1`, 2자리 zero-padded (01~12) |
| `DD` | `date.getDate()`, 2자리 zero-padded (01~31) |
| `요일` | `['일','월','화','수','목','금','토'][date.getDay()]` 고정 배열 참조(로케일 API 미사용) |

### 3.2 시간 표시 형식 (기본 24시간)

```
HH:MM:SS
예: 08:03:07, 23:59:59
```

| 요소 | 규칙 |
|---|---|
| `HH` | `date.getHours()`, 2자리 zero-padded (00~23) |
| `MM` | `date.getMinutes()`, 2자리 zero-padded (00~59) |
| `SS` | `date.getSeconds()`, 2자리 zero-padded (00~59) |

12시간 형식 변환 규칙은 §5 참고.

### 3.3 날짜·시간 동일 원천 원칙

한 번의 렌더 사이클에서 날짜와 시간은 **동일한 단일 `new Date()` 호출 결과**에서 함께 파생되어야 한다(날짜용 `Date`와 시간용 `Date`를 별도로 생성하지 않음). 이는 자정 경계를 넘는 순간 날짜와 시간이 서로 다른 시점 값을 참조해 불일치가 생기는 것을 원천 방지하기 위함이다(§9.1).

## 4. 상태 모델 및 전이 — 시작 / 1초 갱신 / 정지 / 재개

### 4.1 상태 목록

| 상태 | 의미 |
|---|---|
| `running` | 매 1초 tick 마다 `new Date()`를 다시 읽어 화면을 갱신하는 상태 |
| `stopped` | tick 루프(`setInterval`)가 정지되어 화면이 정지 시점 값에 고정된 상태 — 내부적으로 실제 시각은 계속 흐르지만 화면 렌더만 멈춤 |

### 4.2 상태 전이표

| From | To | 트리거 | 부수 효과 |
|---|---|---|---|
| (초기 로드) | `running` | 페이지 로드(`DOMContentLoaded`) | 현재 로컬 시각을 즉시 1회 렌더(날짜+시간) 후 `setInterval(tick, 1000)` 시작. "정지" 컨트롤 활성, 상태 라벨 "동작 중" |
| `running` | `stopped` | "정지" 버튼 클릭 **또는** `Space` keydown(상태가 `running`일 때만) | `clearInterval` 로 tick 루프 즉시 정지. 화면에는 정지 시점의 시:분:초 값이 그대로 고정되어 더 이상 바뀌지 않음. 컨트롤 라벨 "재개"로 전환 |
| `stopped` | `running` | "재개" 버튼 클릭 **또는** `Space` keydown(상태가 `stopped`일 때만) | 정지되어 있던 값에서 이어가지 않고 **재개 시점의 실제 로컬 시각**으로 즉시 재동기화하여 1회 렌더 후 `setInterval(tick, 1000)` 재시작 |

> 정의되지 않은 입력(예: `stopped` 상태에서 12/24 전환 버튼 클릭)은 **금지되지 않는다** — 표시 형식 전환은 상태와 무관하게 항상 허용되며, `stopped` 상태에서 형식을 바꾸면 고정되어 있던 값이 새 형식으로 즉시 재포맷되어 표시된다(§5).

### 4.3 1초 갱신(tick)의 결정론 — 드리프트 방지 원칙

각 tick 은 내부 카운터를 1초씩 증가시키는 방식이 아니라, **매 tick 마다 `new Date()`를 새로 호출**하여 그 실측값을 그대로 포맷팅해 렌더링한다. 이 방식은:

- 브라우저 타이머(`setInterval`)의 실행 지연·스로틀링(백그라운드 탭 등, §9.4)이 발생해도 표시값 자체는 항상 "지금 실제 시각"이므로 드리프트가 누적되지 않는다.
- 사용자가 OS 시스템 시각을 실행 중 수동으로 변경해도 별도 예외 처리 없이 다음 tick 에서 자연스럽게 새 시각이 반영된다(§9.5).

## 5. 12/24시간 전환 규칙

### 5.1 형식 전환 컨트롤

- 트리거: 전용 토글 버튼 클릭 **또는** 키보드 단축키(예: `H`)
- 상태와 무관하게 항상 허용(§4.2 참고) — `running`/`stopped` 어느 상태에서도 즉시 재포맷
- 전환은 **다음 tick 을 기다리지 않고** 즉시 현재 표시값을 새 형식으로 재렌더한다

### 5.2 12시간 형식 변환 규칙

| 24시간 값 | 12시간 표시 |
|---|---|
| `00:MM:SS` (자정) | `오전 12:MM:SS` |
| `01~11:MM:SS` (오전) | `오전 01~11:MM:SS` |
| `12:MM:SS` (정오) | `오후 12:MM:SS` |
| `13~23:MM:SS` (오후) | `오후 01~11:MM:SS` (`24시간값 - 12`) |

> 자정(`00`)과 정오(`12`)는 12시간 형식에서 흔히 실수하는 경계값이므로 위 표를 SSOT 로 삼는다(§9.2).

### 5.3 설정 영속화

- 선택된 형식(`"12"` | `"24"`)은 `localStorage["clock:hourFormat"]`에 저장
- 초기값(저장된 값이 없을 때): `"24"`(24시간 형식)
- 새로고침 시 저장된 값을 복원하여 그 형식으로 최초 렌더(§4.2의 "초기 로드" 전이와 함께 수행)
- 정지/재개 상태 자체는 저장하지 않는 것과 대조적으로, 표시 형식만 영속화 대상이다(§9.3)

## 6. 외부 네트워크 미사용 원칙 (필수 명세)

### 6.1 금지 대상

`clock/` 모듈의 어떤 파일(HTML/CSS/JS)에도 다음이 포함되어서는 안 된다:

| 금지 항목 | 예시 |
|---|---|
| `fetch()` 호출 | `fetch("https://...")`, `fetch("/api/...")` 등 형태 무관 전면 금지 |
| `XMLHttpRequest` | `new XMLHttpRequest()` |
| `WebSocket` | `new WebSocket(...)` |
| `EventSource`(SSE) | `new EventSource(...)` |
| 외부 CDN 리소스 | `<script src="https://...">`, `<link href="https://...">`, CSS `@import url(https://...)` |

### 6.2 허용 대상

- `new Date()` — 로컬 시스템 시각 읽기(유일한 "데이터" 원천)
- `localStorage` — 브라우저 로컬 저장소(네트워크 호출 아님), §5.3 형식 설정 저장에만 사용
- 시스템 폰트, 자체 정의 CSS 변수(외부 CDN 폰트/아이콘 미사용)

### 6.3 검증 방법 (반드시 검증 가능한 형태로 유지)

| 방법 | 절차 | 통과 조건 |
|---|---|---|
| 정적 코드 검사 | `grep -rnE "fetch\(|XMLHttpRequest|WebSocket|EventSource|https?://" clock/*.js clock/*.html` | 매치 0건(자체 상대경로 리소스 제외) — `tests/clock-*.test.js` 에 이 검사를 자동화된 assertion 으로 포함 |

## 8. Acceptance Criteria 매핑 (Given/When/Then, 완료 기준 5항목)

### 8.1 항목별 AC

**AC-1 (표시 규칙 — 로컬 날짜 + HH:MM:SS)**
> **Given** 시계 SPA 페이지가 로드된 상태
> **When** 화면이 렌더링되면
> **Then** 날짜 영역에는 `YYYY-MM-DD (요일)` 형식(§3.1, 예: `2026-07-16 (목)`)으로, 시간 영역에는 `HH:MM:SS` 형식(§3.2, 각 2자리 zero-padded)으로 브라우저의 로컬 시스템 날짜·시각이 표시된다.

**AC-2 (시작 시나리오)**
> **Given** 사용자가 시계 SPA 페이지를 처음 연다
> **When** 페이지 로드(`DOMContentLoaded`)가 완료되면
> **Then** 상태가 즉시 `running`으로 진입하여 현재 로컬 시각이 1회 즉시 렌더되고 1초 tick 루프가 시작되며, "정지" 컨트롤이 활성화된다(§4.2) — 사용자가 별도로 "시작" 버튼을 누를 필요가 없다.

**AC-3 (1초 갱신 시나리오)**
> **Given** 시계가 `running` 상태이다
> **When** 시스템 시각이 1초 경과하면
> **Then** 표시된 시:분:초가 정확히 1초 갱신되어 반영되며, 이 값은 내부 누적 카운터가 아니라 매 tick 시점의 `new Date()` 실측값에서 파생되어(§4.3) 장시간 실행해도 드리프트가 누적되지 않는다.

**AC-4 (12/24시간 전환 시나리오)**
> **Given** 시계가 24시간 형식으로 `08:00:00`을 표시 중이다
> **When** 사용자가 형식 전환 컨트롤(버튼/`H` 키)을 누르면
> **Then** 다음 tick 을 기다리지 않고 즉시 12시간 형식 `오전 08:00:00`으로 재렌더되며(§5.1), 자정은 `오전 12:00:00`, 정오는 `오후 12:00:00`으로 표시되고(§5.2), 이 형식 선택은 `localStorage`에 저장되어 새로고침 후에도 유지된다(§5.3).

**AC-5 (정지/재개 시나리오)**
> **Given** 시계가 `running` 상태이다
> **When** 사용자가 "정지" 버튼을 클릭하거나 `Space` 키를 누르면
> **Then** 상태가 `stopped`로 전이되고 tick 루프가 즉시 정지하며 화면에는 정지 시점의 시:분:초 값이 고정된다(§4.2).
> **그리고 When** 이어서 사용자가 "재개" 버튼을 클릭하거나 `Space` 키를 누르면
> **Then** 상태가 `running`으로 전이되고, 화면은 정지되어 있던 값에서 이어지지 않고 **재개 시점의 실제 로컬 시각**으로 즉시 재동기화되어 렌더된 뒤 1초 tick 루프가 재개된다.

### 8.2 외부 네트워크 미사용 원칙 AC

**AC-NETWORK**
> **Given** `clock/` 모듈 소스 전체(HTML/CSS/JS)
> **When** §6.3의 정적 코드 검사(`grep`) 및 브라우저 DevTools Network 탭으로 실행 중 관찰하면
> **Then** `fetch`/`XMLHttpRequest`/`WebSocket`/`EventSource`/외부 CDN(`https://` 리소스) 호출이 0건이며, 최초 로컬 리소스 로드 이후 추가 네트워크 요청이 발생하지 않고, 그 결과 CORS 에러도 0건이다.

### 8.3 BF-840 수용 기준 ↔ 본 문서 매핑표

| BF-840 수용 기준 | 충족 근거 |

## 9. Edge Case 목록

### 9.1 자정 경계에서 날짜·시간 동시 갱신 불일치 방지

`23:59:59 → 00:00:00` 전이 tick 에서 날짜와 시간을 별도의 `Date` 호출로 계산하면 아주 드물게(밀리초 단위 실행 시차) 날짜는 아직 어제, 시간은 `00:00:00`으로 렌더되는 불일치가 생길 수 있다. §3.3 원칙(단일 `Date` 호출에서 날짜·시간 함께 파생)으로 원천 차단한다.

### 9.2 12시간 형식의 자정/정오 표시

24시간 값 `00`을 단순히 `-12` 하면 음수가 되고, `12`를 그대로 두면 "오전 12시"로 잘못 표시되기 쉽다. §5.2 표를 SSOT 로 삼아 `00→오전 12`, `12→오후 12` 특수 케이스를 명시적으로 처리한다.

### 9.3 정지 상태는 영속화되지 않음 — 새로고침 시 항상 재시작

`stopped` 상태 자체는 `localStorage`에 저장하지 않는다(§5.3에서 형식 설정만 저장). 따라서 사용자가 시계를 정지시킨 채로 새로고침(F5)하면 정지 상태는 유지되지 않고 항상 §4.2 "초기 로드 → `running`" 전이가 다시 발생한다 — 이는 결함이 아니라 의도된 동작이다(시계는 기본적으로 "살아있는" 것이 자연스러운 기본값).

### 9.4 백그라운드 탭에서의 타이머 스로틀링

브라우저(특히 Chrome)는 비활성 탭에서 `setInterval` 실행 주기를 늦추거나 스로틀링할 수 있다. §4.3 원칙(매 tick 마다 `new Date()` 재측정)에 따라 탭이 다시 활성화되면 즉시 정확한 현재 시각으로 따라잡으며 드리프트는 없다 — 다만 비활성 구간 동안 화면이 일시적으로 멈춰 있다가 재활성화 시 시각이 "점프"하는 것처럼 보일 수 있다. 이는 브라우저의 알려진 스케줄링 특성이며 본 스펙의 결함이 아니므로 별도 보정(Web Worker 등) 없이 그대로 둔다(§10 비범위).

### 9.5 실행 중 시스템 시각 수동 변경

사용자가 OS 시스템 시각을 시계 실행 중 수동으로 변경(또는 자동 표준시 조정)하는 경우, 내부에 누적 카운터가 없으므로(§4.3) 다음 tick 에서 별도 예외 처리 없이 자연스럽게 새 시각이 반영된다.

### 9.6 날짜 계산은 전적으로 `Date` 객체에 위임

윤년 2/29, 월말일수, DST(서머타임) 등 달력 계산 로직은 자체 구현하지 않고 전적으로 브라우저 `Date` 객체(§3.1의 getter 들)에 위임한다(Simplicity First — 이미 존재하는 표준 API를 재구현하지 않음).
